"Here's what's different from your current SAS-based world:

Traditional SAS approach:

/* Traditional SAS overnight batch job */
/* Step 1: Extract claims from source system (1 hour) */
PROC SQL;
    CREATE TABLE work.claims_extract AS
    SELECT claim_id, accident_date, lodgement_date,
           incurred_cumulative, paid_cumulative,
           product, state, status
    FROM oracle.claims_transactions
    WHERE transaction_date <= TODAY();
QUIT;

/* Step 2: Calculate development months (30 min) */
DATA work.claims_dev;
    SET work.claims_extract;
    accident_year = YEAR(accident_date);
    dev_month = INTCK('MONTH', accident_date, transaction_date);
RUN;

/* Step 3: Build triangle with PROC TABULATE (2 hours) */
PROC TABULATE DATA=work.claims_dev OUT=work.triangle;
    CLASS product state accident_year dev_month;
    VAR incurred_cumulative paid_cumulative;
    TABLE product*state*accident_year,
          dev_month*(incurred_cumulative*SUM paid_cumulative*SUM);
RUN;

/* Step 4: Export to Excel for CFO (30 min) */
PROC EXPORT DATA=work.triangle
    OUTFILE="/actuarial/triangles/triangle_march_2026.xlsx"
    DBMS=XLSX REPLACE;
RUN;

/* Total runtime: 4-8 hours (overnight batch) */
  • Extract step: PROC SQL pulls from Oracle (1 hour)
  • Transform step: DATA steps and PROC TABULATE build triangle (2-3 hours)
  • Export step: PROC EXPORT to Excel for board packs (30 min)
  • Total runtime: 4-8 hours - must run overnight
  • Frequency: Once per month (too expensive to run daily)
  • Problem: By the time the job finishes, new claims have arrived - triangle is already stale
  • Scalability: As data grows, runtime increases - eventually hits resource limits

Can't answer "what's the reserve right now?" - have to wait for the next scheduled batch run.

Databricks approach:

ORDER BY accident_year, dev_period;
  • Data flow: New claim lands in Bronze → SDP detects → Silver validates → Gold recalculates automatically
  • Runtime: Triangle refreshes in seconds, not hours
  • Query performance: Sub-second response on 37,000 claims (scales to millions)
  • Frequency: Real-time - refreshes as data arrives

The Problem

Pain points:

Solution 1: Embedded Data Quality (DQ as Code)

"Let me show you how data quality works on Databricks. We've embedded 6 DQ expectations directly in the pipeline code. This isn't an external script - it's part of the transformation itself."

[Navigate to SDP Pipeline → Event Log]

Policy Data Quality Rules (3 expectations)

Expectation Rule Business Reason
valid_dates expiry_date > inception_date Policy can't expire before it starts
valid_state state IN ('NSW','VIC','QLD','WA','SA','TAS','ACT','NT') Only Australian states allowed
positive_premium annual_premium > 0 Can't have negative or zero premium

Claims Data Quality Rules (3 expectations)

Expectation Rule Business Reason
incurred_gte_paid incurred_cumulative >= paid_cumulative Fundamental actuarial rule! Can't pay more than incurred
valid_lodgement lodgement_date >= accident_date Can't lodge claim before accident happened
valid_status status IN ('open','closed','reopened') Only valid claim statuses

"Here's what makes this powerful: Let me show you the actual code."

# This is the actual SDP pipeline code

@dlt.table(name="policy_inforce")
@dlt.expect_or_drop("valid_dates", "expiry_date > inception_date")
@dlt.expect_or_drop("valid_state", "state IN ('NSW','VIC','QLD','WA','SA','TAS','ACT','NT')")
@dlt.expect_or_drop("positive_premium", "annual_premium > 0")
def policy_inforce():
    return spark.table("actuary_corpfin.bronze.policy_raw")

@dlt.table(name="claims_transactions")
@dlt.expect_or_drop("incurred_gte_paid", "incurred_cumulative >= paid_cumulative")
@dlt.expect_or_drop("valid_lodgement", "lodgement_date >= accident_date")
@dlt.expect_or_drop("valid_status", "status IN ('open','closed','reopened')")
def claims_transactions():
    return spark.table("actuary_corpfin.bronze.claims_transactions_raw")

"Three key points about this approach:

  1. Version-controlled: This code lives in Git. You can't bypass it. You can't forget to run it. It's part of the pipeline definition.
  2. Always executes: Every time the pipeline runs, these 6 checks execute. 100% consistency.
  3. Action on failure: expect_or_drop means if a record fails, it's automatically dropped from the clean table. You can also use expect_or_fail (stop pipeline) or expect_or_warn (log but continue).

Out of 10,000 policies, we dropped 200 records (2%) that failed validation. Out of 37,437 claims, we quarantined 737 records. The pipeline shows you exactly what failed and why."

Solution 2: Quarantine Pattern (Don't Drop Silently, Investigate)

"Here's a critical distinction. When a record fails validation, we don't just drop it silently. We quarantine it for investigation."

[Navigate to SQL Editor]

SELECT * FROM actuary_corpfin.silver.claims_dq_quarantine
LIMIT 10

"Look at what's in quarantine:

  • 142 claims with lodgement dates in the future - these are data errors from the source system
  • Example: Claim CLM_00012345, lodged on 2027-05-15 - that's 14 months in the future!
  • Each quarantined record includes: claim_id, transaction details, and the reason it failed
# The quarantine table is defined in the SDP pipeline

@dlt.table(name="claims_dq_quarantine")
def claims_dq_quarantine():
    return (
        spark.table("actuary_corpfin.bronze.claims_transactions_raw")
        .filter(F.col("lodgement_date") > F.current_date())
    )

Why quarantine instead of dropping?

  1. Investigation: Is this a data entry error? A system bug? A valid late-reported claim? You need to see the data to decide.
  2. Remediation: Fix the source data, re-run the pipeline, records flow through
  3. Audit trail: Regulator asks "did you drop any claims?" You show them the quarantine table with full transparency
  4. Pattern detection: If you see 142 future-dated claims, that's a systemic issue in the source system you need to fix upstream

You can't investigate what you've silently deleted. Quarantine preserves evidence and gives actuaries visibility into data quality issues."

Solution 2: Variance Analysis Across Quarters (Reserve Movement Explanation)

"Here's another powerful use case. CFO comes to the actuary: 'Our CTP reserve increased $5 million from Q4 2025 to Q1 2026. Which accident years drove this movement? I need to explain it to the board.'"

"In the traditional world, you'd have to:

  1. Find the Q4 2025 Excel triangle file (was it 'Triangle_Q4_Final.xlsx' or 'Triangle_Q4_Final_v2.xlsx'?)
  2. Find the Q1 2026 triangle
  3. Manually copy/paste into a comparison spreadsheet
  4. Calculate differences row by row
  5. Sort to find the biggest movers

This takes hours and is error-prone. On Databricks, it's a single query:"

-- Compare Q1 2026 vs Q4 2025 reserves by accident year
SELECT
    q1.product,
    q1.state,
    q1.accident_year,
    q1.cumulative_incurred as q1_2026_reserve,
    q4.cumulative_incurred as q4_2025_reserve,
    (q1.cumulative_incurred - q4.cumulative_incurred) as reserve_movement,
    ROUND(((q1.cumulative_incurred - q4.cumulative_incurred) / q4.cumulative_incurred * 100), 1) as pct_change
FROM actuary_corpfin.gold.development_triangles
     TIMESTAMP AS OF '2026-03-31T23:59:59Z' q1
JOIN actuary_corpfin.gold.development_triangles
     TIMESTAMP AS OF '2025-12-31T23:59:59Z' q4
    ON q1.product = q4.product
    AND q1.state = q4.state
    AND q1.accident_year = q4.accident_year
    AND q1.dev_period = q4.dev_period
WHERE q1.product = 'CTP'
GROUP BY q1.product, q1.state, q1.accident_year, q1.cumulative_incurred, q4.cumulative_incurred
ORDER BY ABS(reserve_movement) DESC
LIMIT 10;
