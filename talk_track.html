Solution 1: IFRS 17 Cohort Reporting

"Let me show you IFRS 17 cohort reporting. APRA requires us to group policies by inception period."

SELECT cohort_id,
       cohort_inception_year,
       cohort_inception_quarter,
       policy_count,
       gross_written_premium,
       incurred_claims_ytd,
       loss_ratio,
       loss_ratio_flag
FROM actuary_corpfin.gold.ifrs17_cohorts
WHERE loss_ratio_flag = 'ABOVE_100'
ORDER BY loss_ratio DESC
LIMIT 20

"We've created 522 cohorts - grouped by inception quarter, exactly how APRA wants it.

467 cohorts are flagged 'ABOVE_100' - meaning loss ratio over 100%. Those need reserve strengthening, and APRA wants to see them highlighted in your submission.

This table is submission-ready. Export to Excel, attach to APRA portal, done."

Problem 4: Data Quality & Trust 5 minutes

SCREEN: DLT Pipeline + SQL Editor

The Problem

Before the CFO submits numbers to the board, or the actuary submits to APRA, the question is always: Can we trust these numbers?

Pain points:

  • Data errors discovered after submission (embarrassing)
  • DQ checks are external scripts - easy to bypass or forget
  • No visibility into what data failed validation
  • Records silently dropped - you never know what you lost

Solution 1: Embedded Data Quality (DQ as Code)

"We've built 6 DQ expectations in the pipeline as code:"

Policy expectations:

  1. valid_dates: Expiry date must be > Inception date
  2. valid_state: State must be NSW, VIC, QLD, WA, SA, TAS, ACT, NT
  3. positive_premium: Annual premium must be > 0

Claims expectations:

  1. incurred_gte_paid: Incurred must be ≥ Paid (fundamental actuarial rule)
  2. valid_lodgement: Lodgement date must be ≥ Accident date
  3. valid_status: Status must be open, closed, or reopened
@dlt.expect_or_drop("incurred_gte_paid",
                    "incurred_cumulative >= paid_cumulative")

"Version-controlled. Can't be bypassed. Always execute."

Solution 2: Quarantine Pattern (Don't Drop, Investigate)

SELECT * FROM actuary_corpfin.silver.dq_quarantine
LIMIT 10

"If a record fails validation, it goes to quarantine:

  • error_type: "future_lodgement" - claim lodged in the future
  • entity_id: CLM_00012345 - which claim failed
  • error_message: "Lodgement date 2027-05-15 is in the future"

Why quarantine instead of drop? Because actuaries need to investigate: data error? Valid exception? System bug?

You can't investigate what you've deleted. Quarantine preserves evidence."

SELECT MAX(lodgement_date) as latest_claim
FROM actuary_corpfin.bronze.claims_transactions_raw

"Latest claim: March 18, 2026 - that's today. It landed this morning."

SELECT product, state, accident_year,
       SUM(cumulative_incurred) as total_reserve
FROM actuary_corpfin.gold.development_triangles
WHERE product = 'CTP' AND state = 'NSW'
GROUP BY product, state, accident_year
ORDER BY accident_year DESC

"This includes today's claim. The triangle has already refreshed.

How? Claim lands → DLT pipeline detects → Silver validates → Gold recalculates → Actuary queries

No overnight batch. No waiting."

Solution 3: Sub-Second Query Performance

SELECT product, state, accident_year,
       COUNT(*) as claim_count,
       SUM(incurred_cumulative) as total_incurred,
       AVG(development_month) as avg_development,
       MAX(lodgement_date) as latest_lodgement
FROM actuary_corpfin.bronze.claims_transactions_raw
GROUP BY product, state, accident_year
ORDER BY total_incurred DESC

"Execution time: 0.8 seconds

Problem 7: Audit & Historical Reconstruction 3 minutes

SCREEN: Unity Catalog (History tab) + SQL Editor

The Problem

The auditor asks:

  • "What reserve did you hold for Motor NSW in Q2 2025?"
  • "Prove you submitted this exact number to APRA."
  • "Compare your Q4 2025 triangle to your Q1 2026 triangle - what changed?"

In the traditional world: Manual snapshots, re-running jobs, no guarantee of reproducibility

Solution 1: Time Travel - Reconstruct Any Prior State

SELECT * FROM actuary_corpfin.gold.development_triangles
TIMESTAMP AS OF '2025-06-30T23:59:59Z'
WHERE product = 'Motor' AND state = 'NSW'

"This returns exactly what existed on June 30, 2025.

Not a re-run. Not a snapshot. The actual versioned state."

Solution 2: Variance Analysis Across Quarters

SELECT
    q1.accident_year,
    q1.cumulative_incurred as q1_incurred,
    q4.cumulative_incurred as q4_incurred,
    (q1.cumulative_incurred - q4.cumulative_incurred) as movement
FROM actuary_corpfin.gold.development_triangles
     TIMESTAMP AS OF '2026-03-31' q1
JOIN actuary_corpfin.gold.development_triangles
     TIMESTAMP AS OF '2025-12-31' q4
    ON q1.product = q4.product
    AND q1.accident_year = q4.accident_year
WHERE q1.product = 'Motor' AND q1.state = 'NSW'
ORDER BY ABS(movement) DESC

"This shows exactly which accident years drove reserve movements between quarters.

Instant variance analysis. No manual Excel diffs."
